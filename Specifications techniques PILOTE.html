1. Contexte et objectif

PILOTE est un outil local destiné à accompagner les agents d’une MDPH pendant la numérisation de documents. Il répond à un besoin d’orientation : un document numérisé doit ensuite être transmis à l’instructrice concernée.

L’outil permet de reporter les informations disponibles dans MultiGest et sur le document papier, de rechercher l’instructrice dans les fichiers Excel de sectorisation, d’associer le PDF téléchargé et de préparer un classement final.

Positionnement : PILOTE ne remplace pas MultiGest, ne consulte pas ses bannettes et n’y écrit aucune donnée. Il constitue un tableau de suivi local enrichi pendant la numérisation.
Besoin Réponse apportée
Identifier l’instructrice Recherche automatique dans les sectorisations adulte, PCH et enfant.
Suivre un lot important Liste numérotée, recherche, filtres et statistiques en temps réel.
Contrôler les PDF Statut visuel, validation du fichier et contrôle avant export.
Interrompre le travail Sauvegarde et reprise des données de session au format JSON.
Préparer la transmission Classement des PDF par instructrice avec une zone A_VERIFIER.

Hors périmètre

  • connexion directe à MultiGest ;
  • lecture ou déplacement automatique dans les bannettes ;
  • base de données et comptes utilisateurs ;
  • envoi de données vers Internet ;
  • sauvegarde automatique permanente ;
  • restauration actuelle du contenu des PDF depuis le JSON.

5. Fonctionnement technique

5.1 Import et validation des fichiers Excel

L’agent sélectionne les trois fichiers avec un champ de type fichier. Le navigateur transmet leur contenu à PILOTE, puis SheetJS lit les feuilles du classeur.

Fichier attendu Colonnes d’identification
Sectorisation adulteCommune et Assistant de gestion
Sectorisation PCHCommune et Instructeur PCH
Sectorisation enfantNom, Commune et Instructrice enfant

PILOTE recherche les en-têtes dans les premières lignes, ignore les lignes non exploitables et refuse un fichier qui possède les en-têtes sans aucune donnée utile. Après validation, des références de villes, d’écoles et d’instructrices sont construites pour alimenter les suggestions.

5.2 Normalisation des recherches

Les recherches sont comparées après passage en minuscules, suppression des accents et harmonisation des espaces, tirets et apostrophes. Ainsi, ÉVREUX, evreux et une variante de casse correspondent à la même valeur.

5.3 Création de la session

La session est créée uniquement lorsque les trois sectorisations sont prêtes. Son identifiant suit la convention YYYY-MM-DD_AGENT. Les informations sont conservées dans la mémoire de la page avec une liste de documents initialement vide.

5.4 Modèle d’un document

Groupe Données principales
IdentificationIdentifiant interne, nom MultiGest, dates de création et modification.
QualificationPublic, type de document, PCH et hors département.
RechercheVille, école ou GEVASCO.
AttributionInstructrice, correction manuelle, source et statut.
PDFNom, contenu temporaire, présence et besoin de réassociation.

5.5 Attribution de l’instructrice

  • un adulte est recherché par ville dans la sectorisation adulte ;
  • un document PCH est recherché par ville dans la sectorisation PCH ;
  • un enfant est recherché par école et ville, puis par ville si nécessaire ;
  • une école homonyme d’une autre ville n’est pas retenue ;
  • un recours ou un hors département est signalé à vérifier ;
  • une instructrice peut être indiquée manuellement pour un cas particulier.

5.6 PDF

Le fichier sélectionné doit être non vide, avoir une extension PDF ou un type compatible et contenir la signature %PDF- dans son début. Le navigateur fournit un accès temporaire à son contenu tant que la page reste ouverte.

5.7 Exports

Avant l’export, PILOTE vérifie la présence des PDF et les informations obligatoires. Les attributions à vérifier et les doublons déclenchent un avertissement.

Méthode Fonctionnement
Dossiers directs Edge et Chrome utilisent showDirectoryPicker après autorisation de l’agent.
Archive ZIP JSZip construit la même arborescence dans un fichier téléchargé, notamment pour Firefox.

Les fichiers exportés reprennent le nom MultiGest. Les doublons dans un même dossier utilisent _002, _003, etc. Les documents sans attribution certaine sont placés dans A_VERIFIER.

Sauvegarde JSON

Le JSON contient l’agent, la date de tampon, l’identifiant de session et toutes les informations structurées des documents. Il possède un identifiant d’application et une version de format afin de refuser les fichiers incompatibles.

Le JSON conserve le nom et l’état du PDF, mais pas son contenu. Après restauration, l’agent doit actuellement redonner accès aux PDF. Une sauvegarde complète au format ZIP, contenant le JSON et les PDF, est une évolution envisagée pour les sessions volumineuses.

7. Robustesse et sécurité

Sujet Mesure présente
Excel incorrectValidation de l’extension, des colonnes et de la présence de données exploitables.
PDF invalideContrôle de l’extension, de la taille et de la signature interne.
JSON invalideContrôle de la syntaxe, du format, des types et des documents.
DoublonDétection insensible aux accents, à la casse et à l’extension PDF.
Perte de travailAvertissement avant fermeture si la session a changé depuis la sauvegarde.
Export incompletBlocage si un PDF manque et avertissement sur les cas à vérifier.
ÉcrasementCréation d’un nouveau dossier de session suffixé si nécessaire.
ConfidentialitéAucun appel réseau et aucune persistance automatique.
Le code applicatif n’utilise ni fetch, ni requête réseau, ni service distant. Les fichiers sont lus uniquement après une action explicite de l’agent.

Protection des données

L’absence d’envoi Internet ne dispense pas d’appliquer les règles internes de la MDPH. Les PDF, sauvegardes JSON et exports doivent être stockés dans des emplacements autorisés, protégés et supprimés selon les durées de conservation définies.

8.2 Déploiement

  1. Conserver l’ensemble des fichiers et dossiers de PILOTE.
  2. Copier le projet dans un emplacement approuvé par la DNUM.
  3. Créer éventuellement un raccourci vers index.html.
  4. Ouvrir le fichier dans un navigateur compatible.

Aucun exécutable n’est compilé et aucun serveur n’est démarré. Le dossier complet constitue le livrable applicatif. Il peut être distribué dans une archive ZIP puis extrait sur le poste ou sur un emplacement réseau autorisé.

9. Limites et maintenance

9.1 Limites connues

  • la session reste temporaire jusqu’au téléchargement d’un JSON ;
  • le JSON actuel ne contient pas les PDF ;
  • les fichiers Excel doivent être sélectionnés à chaque ouverture ;
  • PILOTE ne contrôle pas la fraîcheur métier des sectorisations ;
  • une attribution automatique dépend de la qualité des données Excel ;
  • les cas particuliers nécessitent toujours une décision humaine ;
  • Firefox ne propose pas l’API de création directe de dossiers utilisée.

9.2 Maintenance

Type Action

9.3 Recette conseillée

  1. Tester l’import des trois sectorisations et les erreurs d’emplacement.
  2. Tester une session adulte, enfant, PCH, recours et hors département.
  3. Tester une école homonyme dans deux villes.
  4. Tester un PDF valide, vide et faussement renommé.
  5. Tester les doublons MultiGest et les suffixes d’export.
  6. Sauvegarder puis restaurer le JSON.
  7. Tester l’export direct dans Edge ou Chrome.
  8. Tester l’export ZIP dans Firefox.
  9. Effectuer une recette avec des agents sur des données fictives ou autorisées.
